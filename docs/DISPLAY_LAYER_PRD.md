# 歌词与下载进度展示页产品方案

## 1. 文档目标

本文档定义 Music Skill 的“歌词和下载进度展示页”产品方案。

这个页面不是完整音乐播放器，而是 OpenClaw 音乐技能的轻展示层，用于承接聊天、打字、语音控制后的状态反馈和持续信息展示。

目标是解决以下问题：

1. 用户发出播放指令后，看不到当前播放状态。
2. 用户发起下载后，不清楚是否开始、进度如何、是否完成。
3. 用户想看歌词时，仅靠聊天气泡展示体验太碎。
4. 语音控制场景下，缺少可视化确认面板。

## 2. 产品定位

### 2.1 页面定位

这是一个轻量状态页，不是重型播放器主界面。

它的职责是：

1. 展示当前正在播放的内容。
2. 展示歌词。
3. 展示下载任务进度。
4. 展示最近一次操作结果。
5. 提供少量高频快捷操作。

### 2.2 与 OpenClaw 的关系

OpenClaw 负责：

1. 接收用户文字和语音指令。
2. 理解意图并调用 Music Skill。
3. 触发展示页刷新或打开展示页。

展示页负责：

1. 把音乐状态持续可视化。
2. 作为语音和聊天命令的视觉反馈面板。
3. 承接需要持续查看的信息，如歌词和下载进度。

## 3. 目标用户

1. 使用 OpenClaw 通过聊天或语音控制音乐的用户。
2. 一边工作一边听音乐、不想频繁切换播放器窗口的用户。
3. 需要实时查看歌词的用户。
4. 需要确认下载是否完成的用户。

## 4. 设计原则

1. 轻量，不做成完整播放器首页。
2. 信息优先，减少无关视觉噪声。
3. 状态连续，打开页面后能立即知道“现在发生了什么”。
4. 聊天驱动，页面是技能的补充而不是替代。
5. 支持桌面端小尺寸显示。

## 5. 适用场景

### 5.1 聊天控制后查看结果

用户说：
`播放录音`

页面展示：

1. 当前歌曲卡片更新为《录音》
2. 播放状态变为“播放中”
3. 歌词区展示歌词或歌词不可用提示

### 5.2 语音控制后确认状态

用户说：
`暂停`

页面展示：

1. 当前状态切换为“已暂停”
2. 进度停留在当前时间点

### 5.3 下载任务查看

用户说：
`下载这首歌`

页面展示：

1. 新增下载任务卡片
2. 显示进度、状态、保存路径
3. 下载完成后给出成功提示

## 6. MVP 范围

第一阶段页面只做以下内容：

1. 当前播放模块
2. 歌词模块
3. 下载任务模块
4. 最近操作反馈模块
5. 少量快捷操作按钮

不在 MVP 内：

1. 完整播放列表页面
2. 复杂歌单编辑
3. 高级均衡器和音效
4. 社交分享
5. 评论、推荐流、发现页

## 7. 页面结构

建议采用单页三段式布局。

## 7.1 顶部：当前播放卡片

展示内容：

1. 封面
2. 歌名
3. 歌手
4. 专辑名
5. 播放状态
6. 当前播放进度
7. 总时长

快捷操作：

1. 播放/暂停
2. 上一首
3. 下一首
4. 收藏
5. 下载

用户价值：

1. 一眼确认当前歌曲
2. 语音操作后有即时视觉反馈

## 7.2 中部：歌词区

展示内容：

1. 当前歌曲歌词
2. 当前行高亮
3. 无歌词时显示空状态

空状态文案建议：

1. `当前歌曲暂无歌词`
2. `歌词加载中`
3. `本地歌曲未匹配到歌词`

MVP 建议能力：

1. 支持静态歌词显示
2. 支持基础滚动定位
3. 支持当前行高亮

后续增强：

1. 双语歌词
2. 点击歌词跳转播放位置
3. 歌词搜索与修正

## 7.3 底部：下载任务区

展示内容：

1. 最近下载任务列表
2. 单个任务名称
3. 当前状态
4. 进度条
5. 保存路径
6. 错误原因

任务状态：

1. 等待中
2. 下载中
3. 已完成
4. 失败

建议默认只显示最近 3 到 5 条任务。

## 7.4 辅助区域：操作反馈条

用于展示最近一次动作结果，例如：

1. `已收藏《录音》`
2. `已把《录音》加入歌单“学习”`
3. `下载完成`
4. `未找到歌词`

展示方式建议：

1. 顶部轻提示
2. 底部状态条
3. 操作后自动消失

## 8. 关键交互流程

## 8.1 播放歌曲

1. 用户通过 OpenClaw 输入 `播放录音`
2. Music Skill 执行播放
3. 页面当前播放卡片更新
4. 歌词区切换到目标歌曲歌词
5. 最近操作反馈显示 `正在播放《录音》`

## 8.2 暂停与继续

1. 用户通过语音说 `暂停`
2. 页面播放状态变为暂停
3. 进度停止更新
4. 用户说 `继续`
5. 页面恢复进度更新

## 8.3 下载歌曲

1. 用户说 `下载这首歌`
2. 页面下载区新增任务
3. 若是 MVP 占位下载，显示“准备中/处理中/已完成”
4. 完成后显示文件路径

## 8.4 歌词不可用

1. 用户播放一首没有歌词的歌
2. 歌词区展示空状态
3. 页面仍保留当前歌曲和下载区，不让整页显得失效

## 9. 数据需求

页面需要以下数据：

### 9.1 当前播放数据

1. trackId
2. title
3. artist
4. album
5. coverUrl
6. playbackStatus
7. progressMs
8. durationMs

### 9.2 歌词数据

1. lyricText
2. timecodedLines
3. source
4. loadStatus

### 9.3 下载任务数据

1. taskId
2. trackTitle
3. artistName
4. progress
5. status
6. filePath
7. errorMessage

### 9.4 操作反馈数据

1. actionType
2. message
3. timestamp

## 10. 页面状态设计

## 10.1 初始状态

页面未开始播放时，展示：

1. `当前没有播放内容`
2. 歌词空状态
3. 下载区为空或显示最近记录

## 10.2 播放中状态

展示：

1. 当前歌曲卡片
2. 播放进度
3. 歌词滚动
4. 最近下载任务

## 10.3 暂停状态

展示：

1. 当前歌曲保持不变
2. 状态显示“已暂停”
3. 进度停止

## 10.4 下载中状态

展示：

1. 下载进度条
2. 任务状态文案
3. 可选取消按钮

## 10.5 失败状态

展示：

1. 明确失败原因
2. 提供可重试提示

## 11. UI 风格建议

因为这是轻展示层，建议采用：

1. 深色半透明卡片
2. 清晰大标题歌名
3. 中等密度歌词排版
4. 简洁直线型进度条
5. 不超过 4 个高频操作按钮

视觉目标：

1. 像“音乐状态面板”
2. 不像“完整播放器首页”
3. 适合停靠在桌面侧边或浮层

## 12. 入口形式建议

可选入口：

1. OpenClaw 内嵌侧边页
2. 独立悬浮窗
3. 小型桌面面板

MVP 推荐：

1. 内嵌侧边页
2. 或固定尺寸悬浮窗

原因：

1. 开发成本低
2. 与聊天界面联动更自然
3. 更适合语音控制后的状态查看

## 13. 与当前 Music Skill 的接口关系

展示页应优先复用当前 Music Skill 已有能力：

1. 当前播放状态
2. 下载任务记录
3. 收藏结果
4. 歌单加入结果

需要新增或增强的接口：

1. 当前播放详细状态接口
2. 歌词查询接口
3. 下载任务列表接口
4. 下载任务状态订阅或轮询接口

## 14. MVP 验收标准

### 14.1 基础验收

1. 用户播放歌曲后，页面能正确显示当前歌曲。
2. 用户暂停后，页面状态能同步更新。
3. 用户发起下载后，页面能显示下载任务记录。
4. 页面能显示歌词或歌词缺失状态。
5. 收藏和加入歌单后，页面能展示操作反馈。

### 14.2 体验验收

1. 用户在 2 秒内看懂当前状态。
2. 下载和播放状态更新清晰。
3. 页面不会干扰聊天主流程。

## 15. 后续迭代方向

1. 显示封面和专辑背景图
2. 支持逐行歌词同步
3. 支持迷你模式
4. 支持最近播放和历史记录
5. 支持点击歌词跳转
6. 支持下载详情页
7. 支持歌单切换和队列可视化

## 16. 一句话结论

歌词和下载进度展示页应该做，而且应该先做成一个轻量状态面板，用来承接 OpenClaw 聊天与语音控制后的可视化反馈，而不是一上来做完整音乐播放器页面。
