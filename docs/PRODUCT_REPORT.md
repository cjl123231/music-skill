# Music Skill 产品分析报告

> 报告日期：2026-03-27  
> 分析对象：Music Skill v0.2.0（Local Edition）

---

## 1. 产品概述

**Music Skill** 是一款面向 [OpenClaw](https://openclaw.ai) 平台的可安装音乐控制插件，当前版本定位为 **Local Edition**（本地优先版）。

其核心价值主张是：

> 让用户无需打开播放器窗口，通过聊天、打字或语音，即可完成本地音乐的播放、收藏、歌单管理与下载。

### 关键特征

| 属性 | 值 |
|---|---|
| 当前版本 | 0.2.0 |
| 技术栈 | TypeScript（ESM）、Node.js 22+、SQLite、pnpm |
| 支持平台 | Windows（完整支持）、macOS（MVP 支持） |
| 存储方案 | SQLite（默认）/ JSON（兜底） |
| 外部依赖 | 无需外部模型 API、无需在线音乐平台 API |
| 输入方式 | 文字、浏览器语音、Windows 耳机麦克风语音 |

## 2. 版本演进

### v0.1.0：脚手架阶段

- 建立 OpenClaw 插件注册结构
- 初步定义 `openclaw.plugin.json` 配置项（`defaultProvider`、`enableDownload`、`enableVoice`）
- 搭建基础 TypeScript 项目框架

### v0.2.0：首个可用本地产品（当前版本）

0.2.0 是从"脚手架"蜕变为"真正可用产品"的关键版本，主要交付：

- **本地音乐库扫描**：支持 `.mp3`、`.flac`、`.wav`、`.m4a`、`.aac`、`.ogg`，自动读取音频元数据
- **本地播放控制**：播放、暂停、继续、上一首、下一首
- **SQLite 持久化**：默认存储迁移至 SQLite，保留 JSON 作为降级方案
- **跨平台播放**：Windows（PowerShell 播放宿主）+ macOS（`afplay` MVP 路径）
- **Windows 耳机语音控制**：唤醒词 `音乐控制`，支持实时麦克风指令
- **浏览器面板**：`panel.html` + `app.js`，提供可视化播放状态与下载列表
- **收藏与歌单**：本地存储收藏记录、歌单管理
- **下载记录**：本地文件复制 + 任务状态追踪

## 4. 技术架构

项目采用清晰的六层分层架构：

```
┌──────────────────────────────────────────────────────┐
│  输入层（Input Layer）                                 │
│  OpenClaw 聊天 / 面板文字 / 浏览器语音 / 耳机语音      │
├──────────────────────────────────────────────────────┤
│  理解层（Understanding Layer）                         │
│  Intent 路由 / Slot 提取 / 上下文管理 / 同义词规范化   │
├──────────────────────────────────────────────────────┤
│  能力层（Capability Layer）                            │
│  播放控制 / 收藏 / 歌单 / 下载 / 查询                  │
├──────────────────────────────────────────────────────┤
│  Provider 适配层                                       │
│  LocalMusicProvider / PlaybackController（跨平台工厂） │
├──────────────────────────────────────────────────────┤
│  存储层（Storage Layer）                               │
│  SQLite（favorites / playlists / downloads / context）│
├──────────────────────────────────────────────────────┤
│  呈现层（Presentation Layer）                          │
│  文本回复 / 浏览器面板 / 下载列表                       │
└──────────────────────────────────────────────────────┘
```

### 核心设计亮点

1. **Provider 抽象**：业务层依赖统一 `MusicProvider` 接口，不绑定具体平台，为后续在线音乐接入预留扩展点。
2. **PlaybackController 工厂**：`create-playback-controller.ts` 按平台（`win32` / `darwin`）分发具体控制器，保持 Provider 层平台无感。
3. **双存储策略**：SQLite 作为默认，JSON 作为降级，并提供迁移脚本 `migrate-json-to-sqlite.ts`。
4. **25个标准意图**：从 `music.play` 到 `music.queue.clear`，覆盖完整播放管理链路。

### 目录结构

```
src/
  plugin/         # OpenClaw 插件入口 & 工具定义
  application/    # 意图路由、Use Case、响应生成
  domain/         # 核心领域实体（Track / Playlist / DownloadTask 等）
  infrastructure/ # Provider 适配、播放控制、存储、下载
  interfaces/     # HTTP 接口、OpenClaw 接口
  shared/         # 错误码、公共类型、工具函数
ui/               # 浏览器面板（HTML + JS + CSS）
scripts/          # 安装、启动、迁移脚本（.ps1 / .sh）
tests/
  unit/
  integration/
```

## 5. 用户价值

### 目标用户

- 已安装 OpenClaw 的桌面端用户
- 边工作边听音乐、不想切换窗口的用户
- 偏好语音控制的用户（尤其佩戴耳机时）
- 希望统一管理本地音乐资产的用户

### 核心用户场景

| 场景 | 操作路径 | 操作轮次 |
|---|---|---|
| 播放本地歌曲 | 说"播放 XXX" → 直接开始播放 | 1 轮 |
| 暂停/切歌 | 说"暂停"/"下一首" | 1 轮 |
| 收藏当前歌曲 | 说"收藏这首歌" | 1 轮 |
| 加入指定歌单 | 说"把这首歌加入学习歌单" | 1 轮 |
| 下载并查看进度 | 说"下载这首歌" → 说"下载好了没" | 2 轮 |
| 耳机远程控制 | "音乐控制，暂停" | 1 轮（免手操作） |

## 7. 已知局限与风险

### 当前局限

| 局限项 | 说明 |
|---|---|
| macOS 音量控制 | 音量变更需重启 `afplay` 进程，体验粗糙 |
| macOS 无宿主级语音 | 仅浏览器内语音，无耳机麦克风支持 |
| 语言理解能力 | 规则驱动，覆盖词条需手动扩展 |
| 无在线内容 | 不支持在线搜索、流媒体播放 |
| SQLite 实验性警告 | Node 内置 SQLite 在运行时输出 experimental warning |
| 无云同步 | 收藏、歌单数据仅存本地 |

### 风险识别

1. **意图识别覆盖率**：口语表达多样，规则引擎难以穷举，可能导致指令失败率上升。
2. **平台兼容性**：macOS 环境下 `openclaw` CLI 路径差异可能导致安装失败。
3. **音频编解码**：部分本地文件可能因系统支持情况不同而无法播放。

## 8. 路线图

### Phase 1（当前 v0.2.0，已完成）

- 本地音乐库 + 播放控制
- SQLite 持久化
- Windows 耳机语音
- macOS MVP 播放路径
- 浏览器面板

### Phase 2（建议方向）

- 增强 macOS 支持（`afplay` 音量改善 / 宿主级语音）
- 扩展本地口语指令覆盖率
- 歌词本地展示
- 更好的面板播放进度与元数据展示

### Phase 3（长期演进）

- 可选接入外部模型 API（增强自然语言理解）
- 可选接入在线音乐平台
- 跨设备同步
- 多步 Agent 音乐推荐行为

### 决策规则

> 若某功能依赖云端，先问：**"本地版能否在不依赖该功能的前提下，依然交付核心用户价值？"** 若能，则不纳入 MVP。

## 10. 总结

Music Skill v0.2.0 完成了从"项目脚手架"到"可用本地产品"的关键跨越。在无需任何外部 API 的前提下，它为 OpenClaw 用户提供了完整的本地音乐控制闭环——播放、收藏、歌单、下载四个核心维度全部可用，Windows 用户还拥有耳机语音控制能力。

当前最突出的局限是：**规则驱动的语言理解难以覆盖口语多样性**，以及 **macOS 支持尚不完整**。这两个方向也是 Phase 2 最值得优先投入的领域。

产品的架构设计前瞻性较好——Provider 抽象和 PlaybackController 工厂为后续扩展（在线音乐、模型增强、多平台）留下了清晰的扩展缝隙，不需要推倒重建核心逻辑。
