# Music Skill 产品文档

## 1. 产品概述

### 1.1 产品名称
Music Skill

### 1.2 产品定位
Music Skill 是面向 OpenClaw 用户的音乐控制技能。用户可以通过与 OpenClaw 聊天，使用文字、语音等自然交互方式，完成音乐播放器常见操作，包括播放音乐、搜索音乐、暂停、切歌、调节音量、收藏、创建歌单、下载等。

### 1.3 产品愿景
让用户不需要学习复杂播放器界面或命令，只需像和助手说话一样，就能自然完成完整的音乐收听与管理流程。

### 1.4 产品目标
1. 让已安装 OpenClaw 的用户可以通过对话直接控制音乐播放。
2. 提供覆盖主流音乐播放器核心能力的自然语言交互体验。
3. 支持文字与语音两种主要输入方式。
4. 为后续接入不同音乐源、账号体系、推荐能力打下统一技能层基础。

## 2. 背景与机会

传统音乐播放器主要依赖按钮、列表和搜索框，操作路径固定。OpenClaw 具备聊天、打字、语音等交互入口，如果能把音乐能力封装成 skill，用户可以用更自然的方式完成操作，例如：

- “播放周杰伦的晴天”
- “把这首歌加入我喜欢”
- “下载这张专辑”
- “音量调小一点”
- “继续刚才那首歌”

这类需求天然适合由对话式技能承接。产品价值在于把播放器能力从 UI 驱动转为意图驱动。

## 3. 用户与使用场景

### 3.1 目标用户
1. 已安装 OpenClaw 的桌面端用户。
2. 有边听音乐边工作、学习、浏览需求的轻度到中度音乐用户。
3. 更偏好语音控制、懒得切窗口操作播放器的用户。
4. 希望统一通过助手管理本地与在线音乐资源的用户。

### 3.2 典型场景
1. 工作时不切换窗口，通过聊天或语音完成播放控制。
2. 听到喜欢的歌时，快速收藏、加入歌单或下载。
3. 想听某个歌手、专辑、风格时，直接口语化搜索和播放。
4. 使用耳机或远离键盘时，通过语音进行暂停、下一首、调音量等操作。
5. 管理个人音乐资产，如喜欢列表、最近播放、下载记录。

## 4. 核心问题

当前音乐控制存在以下问题：

1. 操作依赖播放器窗口，切换成本高。
2. 用户需要适应不同平台和播放器的交互方式。
3. 语音助手通常只支持少量固定指令，不支持复杂音乐管理动作。
4. 搜索、播放、收藏、下载等能力分散，缺少统一入口。

## 5. 产品范围

### 5.1 MVP 范围
MVP 先覆盖“可用且高频”的音乐播放与管理能力：

1. 搜索并播放歌曲、歌手、专辑、歌单。
2. 播放控制：播放、暂停、继续、上一首、下一首、停止。
3. 播放队列控制：查看当前播放、加入下一首、清空队列。
4. 音量控制：调高、调低、静音、设置指定音量。
5. 收藏能力：收藏歌曲、取消收藏、查看我喜欢的音乐。
6. 下载能力：下载单曲、专辑，查看下载进度与结果。
7. 基础歌单能力：创建歌单、加入歌单、播放歌单。
8. 输入方式：文字输入、语音输入。
9. 基础反馈：播放结果、失败原因、歧义澄清。

### 5.2 非 MVP 范围
以下内容不作为第一阶段必须交付：

1. 复杂推荐系统，如每日推荐、猜你喜欢。
2. 多端同步。
3. 社交能力，如分享、评论、共同歌单。
4. 高级音效、均衡器、歌词海报等增强功能。
5. 完整版权和支付体系。
6. 多音乐平台深度打通的高级会员能力。

## 6. 产品能力设计

### 6.1 意图分类
Music Skill 需要识别以下核心用户意图：

1. 播放意图
   例：“播放林俊杰的歌”
2. 搜索意图
   例：“找一首包含稻香的歌”
3. 控制意图
   例：“暂停”“下一首”“从头播放”
4. 收藏意图
   例：“收藏这首歌”
5. 下载意图
   例：“下载这张专辑”
6. 歌单管理意图
   例：“建一个跑步歌单”
7. 查询意图
   例：“现在播放的是什么”“我最近下载了什么”

### 6.2 支持的自然语言能力
系统应支持自然口语、简短命令和上下文连续对话：

1. 直接命令
   例：“播放夜曲”
2. 带条件命令
   例：“播放周杰伦最热门的歌”
3. 上下文引用
   例：“把这首加入收藏”“下载刚才那张专辑”
4. 模糊表达
   例：“放点轻松的音乐”
5. 歧义澄清
   例：搜索到多个同名歌曲时，让用户选择版本

### 6.3 语音交互要求
1. 支持短语音快速指令。
2. 语音转文字后走统一意图识别链路。
3. 对嘈杂环境和口语表达具备一定容错。
4. 支持语音确认与语音反馈，可由 OpenClaw 或宿主能力决定最终表现形式。

## 7. 功能需求

### 7.1 搜索与播放
功能描述：
用户通过文字或语音搜索音乐内容并发起播放。

功能点：
1. 搜索歌曲、歌手、专辑、歌单。
2. 支持精确搜索与模糊搜索。
3. 当结果唯一时直接播放。
4. 当结果不唯一时给出候选项供用户确认。
5. 支持“播放第一首”“播放第三个结果”等后续操作。

示例：
- “播放告白气球”
- “播放 Taylor Swift 的专辑 1989”
- “找一首粤语老歌”

### 7.2 播放控制
功能点：
1. 播放
2. 暂停
3. 继续播放
4. 停止播放
5. 下一首
6. 上一首
7. 单曲循环、列表循环、随机播放
8. 查看当前播放状态

示例：
- “暂停一下”
- “下一首”
- “切到随机播放”
- “现在在播什么”

### 7.3 队列管理
功能点：
1. 把歌曲加入播放队列。
2. 插入为下一首播放。
3. 查看接下来播放列表。
4. 删除队列中的某一首歌。
5. 清空队列。

示例：
- “把青花瓷加入队列”
- “这首下一首播”
- “清空待播放列表”

### 7.4 收藏与喜欢
功能点：
1. 收藏当前歌曲。
2. 取消收藏。
3. 查看喜欢的音乐列表。
4. 播放我喜欢的音乐。

示例：
- “收藏这首歌”
- “取消喜欢”
- “播放我喜欢的歌”

### 7.5 歌单管理
功能点：
1. 创建歌单。
2. 重命名歌单。
3. 添加歌曲到歌单。
4. 从歌单中移除歌曲。
5. 播放指定歌单。
6. 查看歌单内容。

示例：
- “创建一个深夜歌单”
- “把这首歌加入深夜歌单”
- “播放我的健身歌单”

### 7.6 下载管理
功能点：
1. 下载当前歌曲。
2. 下载指定歌曲或专辑。
3. 查看下载状态。
4. 查看下载目录或下载列表。
5. 下载失败时给出原因。

示例：
- “下载这首歌”
- “把这张专辑下载下来”
- “刚才下载成功了吗”

### 7.7 音量与设备控制
功能点：
1. 音量增加或减少。
2. 设为指定音量值。
3. 静音与取消静音。
4. 可选支持切换输出设备。

示例：
- “音量小一点”
- “调到 30%”
- “静音”

## 8. 关键交互设计

### 8.1 基本交互流程
1. 用户发起文字或语音输入。
2. OpenClaw 将输入传给 Music Skill。
3. Skill 识别用户意图与关键实体。
4. 调用音乐能力层执行操作。
5. 将结果组织成自然语言反馈给用户。
6. 如存在歧义或失败，进入追问或纠错流程。

### 8.2 歧义处理
当命中多个结果时，系统应：

1. 优先根据上下文和历史偏好缩小范围。
2. 给出最多 3 到 5 个候选项，避免信息过载。
3. 支持用户说“第一个”“第二个版本”等继续选择。

示例：
用户：“播放演员”
系统：“找到了多个结果，你想播放：
1. 薛之谦《演员》
2. 李荣浩《演员和歌》
3. 纯音乐版本《演员》
你可以回复序号或直接说歌手名。”

### 8.3 失败反馈
失败反馈需要清晰具体，避免只返回“失败”。

示例：
1. “没有找到匹配的歌曲，是否要我搜索歌手或相似名称？”
2. “这首歌当前无法下载，可能受版权限制。”
3. “当前没有正在播放的歌曲，所以无法收藏。”

### 8.4 上下文连续对话
系统应维护最近一次播放对象、搜索结果、当前歌曲、当前专辑、当前歌单等上下文，支持连续指令。

示例：
1. “播放周杰伦的晴天”
2. “收藏这首”
3. “再下载这张专辑”

## 9. 信息架构与模块划分

建议将产品拆为以下模块：

1. 输入层
   负责接收文字、语音输入。
2. 意图理解层
   负责识别播放、收藏、下载等意图和实体。
3. 对话管理层
   负责上下文维护、歧义澄清、多轮交互。
4. 音乐能力层
   负责搜索、播放控制、收藏、歌单、下载等能力编排。
5. 数据层
   负责用户偏好、播放历史、收藏记录、下载记录。
6. 反馈层
   负责将结果以文本或语音形式返回给 OpenClaw。

## 10. 业务规则与约束

1. 若没有明确音乐源，系统需要预留统一 Provider 接口，避免产品和单一平台耦合。
2. 下载能力需要明确版权限制、账号权限和可用范围。
3. 收藏、歌单等用户数据需要绑定用户身份。
4. 涉及删除、清空、批量下载等动作时，建议增加确认机制。
5. 对语音误识别导致的高风险操作，应有二次确认策略。

## 11. MVP 用户故事

1. 作为用户，我希望说“播放某首歌”后，系统能直接开始播放。
2. 作为用户，我希望说“暂停”或“下一首”时，不需要切到播放器窗口。
3. 作为用户，我希望在听到喜欢的歌时，能直接说“收藏这首歌”。
4. 作为用户，我希望说“下载这首歌”后，系统告诉我下载是否开始、是否成功。
5. 作为用户，我希望面对多个搜索结果时，系统能给出清晰候选并支持继续选择。
6. 作为用户，我希望使用语音时，也能和文字一样完成完整控制链路。

## 12. 验收标准

### 12.1 功能验收
1. 用户可通过文字输入完成搜索并播放歌曲。
2. 用户可通过语音输入完成播放、暂停、下一首、收藏、下载等核心动作。
3. 系统能识别并处理至少一轮上下文指代，如“这首歌”“刚才那个专辑”。
4. 遇到多结果时，系统能返回候选项并接受用户继续选择。
5. 收藏、歌单、下载操作有明确结果反馈。

### 12.2 体验验收
1. 常见指令回复清晰，不需要用户理解技术术语。
2. 多数高频操作在 1 轮对话内完成。
3. 歧义和失败场景下，系统提供可继续的下一步建议。

## 13. 关键指标

MVP 阶段建议关注以下指标：

1. 播放指令成功率。
2. 意图识别准确率。
3. 首次响应时延。
4. 语音指令成功率。
5. 收藏与下载功能使用率。
6. 多轮澄清后的任务完成率。
7. 用户 7 日留存与日均使用次数。

## 14. 风险与挑战

1. 音乐源接入复杂，搜索、播放、下载能力可能因平台差异而不一致。
2. 下载与版权限制会直接影响产品完整度。
3. 语音识别错误会放大用户挫败感。
4. 模糊搜索和多结果选择处理不好，会显著降低对话效率。
5. 若 OpenClaw 宿主侧缺少稳定播放器控制接口，Skill 体验会受限。

## 15. 后续迭代方向

1. 个性化推荐，如基于时间、场景、历史偏好的音乐推荐。
2. 更强的语义理解，如“来点适合下雨天听的中文歌”。
3. 歌词显示与歌词跳转。
4. 定时关闭、睡眠模式、番茄钟联动。
5. 多平台账号打通。
6. 本地音乐库与在线音乐统一管理。
7. 桌面浮层、快捷指令、状态栏控制。

## 16. 一句话总结

Music Skill 要解决的核心问题，是把传统音乐播放器的按钮式操作，升级为 OpenClaw 中可通过聊天、打字、语音完成的自然语言音乐控制体验，并优先做出一个覆盖“播放、控制、收藏、下载”的 MVP 闭环。
